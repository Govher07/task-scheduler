Section 1: Data Model                  
                                                                                                                                                           
  Three core entities plus a join/grouping concept:                                                                                                        
                               
  Task                                                                                                                                                     
  - id (UUID)                            
  - name (String) - e.g., "Go to the gym"                                                                                                                  
  - goalId (UUID, nullable) - links to a Goal, null if standalone                                                                                          
  - priority (enum: low, medium, high)                           
  - deadline (DateTime, nullable)                                                                                                                          
  - estimatedDuration (Duration, nullable)
  - effortLevel (enum: low, medium, high) - energy required                                                                                                
  - isCompleted (bool)                                                                                                                                     
  - createdAt, updatedAt (DateTime)                                                                                                                        
                                         
  Event                                                                                                                                                    
  - id (UUID)                            
  - name (String) - e.g., "Daily gym session"
  - taskId (UUID, nullable) - optional link to a Task
  - startTime (DateTime)                                                                                                                                   
  - endTime (DateTime)                                                                                                                                     
  - isRepeating (bool)                                                                                                                                     
  - recurrenceRule (String, nullable) - e.g., RRULE format for repeating events                                                                            
  - createdAt, updatedAt (DateTime)                                            
                                                                                                                                                           
  Goal (replaces "Task Group" from your docs)
  - id (UUID)                                                                                                                                              
  - name (String) - e.g., "Get healthier"
  - description (String, nullable)                                                                                                                         
  - tasks (derived relationship - list of Tasks where goalId matches)
  - createdAt, updatedAt (DateTime)                                                                                                                        
                                                                                                                                                           
  Key decisions:             
  - Event.taskId is nullable - events can be standalone (dentist appointment) or linked to a task                                                          
  - Task.goalId is nullable - tasks can exist without belonging to a goal                        
  - Recurrence uses RRULE strings - standard format (used by Google Calendar, iCal), avoids reinventing recurrence logic                                   
  - Renamed "Task Group" to Goal for clarity since your docs use both terms interchangeably

  Key architectural patterns:                                                                                                                              
                                                                                                                                                           
  - Repository pattern - abstract interfaces in core/repositories/, with Drift implementations now. Swap in Supabase implementations later without touching
   feature code.                                                                                                                                           
  - Drift for local DB - generates type-safe Dart from SQL table definitions, handles migrations                                                           
  - Freezed for models - immutable data classes with copyWith, JSON serialization, equality
  - GoRouter for navigation - declarative routing with a ShellRoute for the bottom nav bar                                                                 
  - Feature providers depend only on core repositories - no cross-feature dependencies

Section 4: Recommender Logic                                                                                                                             
                                         
  For V1, the recommender uses a simple scoring algorithm:                                                                                                 
                                                                                                                                                           
  Score = (priority weight) + (deadline urgency weight)
                                                                                                                                                           
  - Priority weight: High = 30, Medium = 20, Low = 10
  - Deadline urgency: Based on hours remaining until deadline                                                                                              
    - Overdue: +50                                                                                                                                         
    - Due within 24h: +40                                                                                                                                  
    - Due within 3 days: +25                                                                                                                               
    - Due within 7 days: +15                                                                                                                               
    - Due later or no deadline: +0                                                                                                                         
                                                                                                                                                           
  Rules:                                 
  - Only considers incomplete tasks                                                                                                                        
  - Tasks with no deadline are scored on priority alone
  - Ties broken by createdAt (older tasks first)                                                                                                           
  - "Skip" temporarily excludes a task for that session
                                                                                                                                                           
  This keeps it simple and deterministic for V1. The future AI chatbot feature (P2) from your roadmap could replace this with something more sophisticated 
  later.
